# EdgeMesh

A Go service mesh sidecar for Kubernetes workloads running on flaky edge
networks. Each pod runs an `edgemesh` sidecar that owns the outbound RPC
path: gRPC client multiplexing, active health checking, retry with
classified backoff, round-robin or least-pending load balancing, and a
control surface for operators (`meshctl`).

The defaults are tuned for the edge profile: tens of peers, variable
inter-node latency, frequent asymmetric partitions, whole-node dropouts
that last minutes. See [`docs/edge-vs-cluster.md`](docs/edge-vs-cluster.md)
for the comparison with the Envoy / Istio defaults.

## Status

- Go 1.22, gRPC 1.66, distroless image (amd64 + arm64)
- 12-node chaos suite committed, run on every CI build
- `~3.2k` LOC, `~75%` line coverage on `internal/`
- MIT licensed

## How EdgeMesh fits with the rest of the SAY-5 stack

| Repo                                                                     | Scope                                                                                                                                                |
|--------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------|
| [`SAY-5/edgemesh`](https://github.com/SAY-5/edgemesh) (this repo)        | Sidecar process. Per-pod gRPC mesh data plane. Health checking, retry, LB, chaos handling. Talks to peer sidecars over the wire.                     |
| [`SAY-5/inference-router`](https://github.com/SAY-5/inference-router)    | Single-process C++ router for in-host model inference. No mesh, no peers, no health probing. Routes calls between local model workers on one box.    |
| [`SAY-5/kube-deploy`](https://github.com/SAY-5/kube-deploy)              | Provisioning CLI. Stamps out Kubernetes clusters and applies manifests. Does not run at request time. Does not know about RPC, health, or retries.   |

The three repos do not overlap. `edgemesh` is a data plane that runs in
every pod. `inference-router` is a single-host process that never opens a
network socket. `kube-deploy` is a build-time tool that places the
manifests `edgemesh` ships in `k8s/`.

## 12-node chaos result

The numbers below come from
[`bench/results/chaos-12-node.json`](bench/results/chaos-12-node.json),
generated by `cmd/chaos-report` against the harness in
`tests/integration`.

| Metric                              | Value             |
|-------------------------------------|-------------------|
| Nodes                               | 12                |
| Scenarios run                       | 200               |
| Scenarios passed                    | 200 / 200         |
| Total RPCs                          | 16,000            |
| RPCs succeeded                      | 15,764            |
| RPCs failed (classified)            | 236               |
| RPCs unaccounted                    | 0                 |
| LB invariant violations             | 0                 |
| Convergence p50 / p95 / max         | 1 / 8 / 9 ms      |
| Wall clock                          | 1,369 ms          |
| Scenario kinds                      | baseline, node_down, partition_pair, partition_star, random_drop, recovery |

Assertions the suite checks per scenario:

1. The load balancer never selects a peer that is unhealthy at pick time.
2. Every issued RPC is accounted for as a success or a classified failure.
3. The mesh converges to the expected health view within 2 s.
4. The system returns to all-healthy within 2 s when the scenario is cleared.

Methodology: [`docs/chaos-methodology.md`](docs/chaos-methodology.md).

## Build

```
make build           # bin/sidecar and bin/meshctl
make test            # unit tests with race detector, coverage
make test-chaos      # full 200-scenario 12-node chaos run
make bench-smoke     # 1-iteration smoke of all benchmarks
make docker          # local image
```

## Run

Sidecar:

```
./bin/sidecar -config scripts/compose-config-a.yaml
```

`meshctl` against a running sidecar:

```
./bin/meshctl status -addr 127.0.0.1:8080
./bin/meshctl drain  -addr 127.0.0.1:8080 -grace 30
./bin/meshctl health -addr 127.0.0.1:8080
```

Three-node compose harness:

```
docker compose up --build
./bin/meshctl status -addr 127.0.0.1:8080
./bin/meshctl status -addr 127.0.0.1:8081
./bin/meshctl status -addr 127.0.0.1:8082
```

## Kubernetes

`k8s/base` is a Kustomize base with one Deployment (the app and the
sidecar share a Pod), one ConfigMap, and one Service. The overlays under
`k8s/overlays/{dev,stg,prod}` set namespace, prefix, replica count, and
labels. `kubeconform` validates every rendered overlay in CI.

```
kubectl apply -k k8s/overlays/dev
kubectl apply -k k8s/overlays/stg
kubectl apply -k k8s/overlays/prod
```

## Configuration

YAML, validated at startup. Example with the defaults:

```yaml
node_id: node-a
listen:
  address: 0.0.0.0:8080
load_balancer: round-robin    # or "least-pending"
health:
  interval_ms: 1000
  timeout_ms: 200
  healthy_to_unhealthy: 3
  unhealthy_to_healthy: 2
retry:
  max_attempts: 3
  base_ms: 50
  multiplier: 4
  max_ms: 1000
  jitter_frac: 0.2
services:
  - name: echo
    peers:
      - id: peer-a
        address: echo-a.default.svc.cluster.local:8080
      - id: peer-b
        address: echo-b.default.svc.cluster.local:8080
    methods:
      - name: Say
        idempotent: true
```

Non-idempotent methods never retry, regardless of the gRPC status code.
