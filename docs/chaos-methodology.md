# Chaos Methodology

This document describes how the 12-node chaos suite works and what it
verifies. The headline numbers in the README come from running this suite
and committing the resulting JSON.

## Topology

The simulator runs 12 sidecars and 12 services inside a single Go test
process. Every node is a peer of every other node (full mesh, fan-out 11).
Each "wire" call goes through `chaos.FaultInjector.Deliver` which decides
whether to deliver, drop, or delay the message.

No real sockets are involved. The deliberate trade-off:

- We give up testing the TCP / gRPC wire path under chaos. That path is
  covered by the unit tests in `internal/transport`.
- We gain a deterministic, fast, hermetic test that exercises the
  interaction between health checker, LB, and retry under realistic fault
  patterns at a scenario rate (200/s on the developer laptop) that real
  Kubernetes chaos tools cannot match.

## Scenario taxonomy

The harness draws scenarios uniformly from six kinds:

| Kind                | What it does                                          |
|---------------------|-------------------------------------------------------|
| `baseline`          | no faults; verifies steady-state RPCs succeed         |
| `node_down`         | one random node fully unreachable in both directions  |
| `partition_pair`    | one directed link severed (a -> b drops, b -> a OK)   |
| `partition_star`    | one node's outbound links all severed                 |
| `random_drop`       | one transient partition; exercises the retry path     |
| `recovery`          | no faults; verifies the system stays healthy when idle|

Each scenario runs `RPCsPerScenario` concurrent RPCs (default 80) from
random healthy source nodes. Between scenarios the harness clears all
faults and drives recovery to a known-good state.

## Assertions (per scenario)

1. **Safety**: the LB never selects a peer it observed as unhealthy.
2. **Liveness**: every issued RPC ends in either success or a classified
   error. No RPC is lost without a status.
3. **Convergence**: every healthy node's view of every other node matches
   the expected health map within `ConvergeBudget` (2s default). Downed
   nodes' views are excluded; a downed node cannot probe its peers.

## Committed result

`bench/results/chaos-12-node.json` is generated by `go run
./cmd/chaos-report` with `Scenarios=200, RPCsPerScenario=80, Seed=1`.
The file is committed and cited verbatim in the README. CI does not
regenerate it; CI runs a smaller smoke run (`CHAOS_SCENARIOS=60`) to keep
the runner fast.

## What this does *not* test

- Wire-level gRPC behaviour under partial reads, partial writes, or
  malformed frames. The standard gRPC implementation handles those.
- TLS / mTLS. Out of scope for this build.
- Multi-region latency profiles. The simulator uses a single uniform
  latency distribution.
- Real Kubernetes scheduling. The K8s manifests under `k8s/` are
  validated structurally (kubeconform) but never applied in CI.
